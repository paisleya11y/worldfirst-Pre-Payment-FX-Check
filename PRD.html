01 / BACKGROUND

项目背景

1.1 WorldFirst 在蚂蚁国际四象限中的定位

蚂蚁国际通过四个产品板块 + 一套全球支付网络，覆盖跨境商业场景的不同环节。WorldFirst 是其中面向 B2B 企业端的板块。

把这套框架按"面向谁 / 核心角色 / 价值链位置"三个维度展开，能更清楚地看到 WorldFirst 与其他板块的边界：

WorldFirst 在集团里的战略意义：它把蚂蚁国际从"帮商家收消费者的钱"，延伸到"帮商家经营全球资金"。支付是一笔交易，资金管理是一个商家的经营生命周期——只要商家的收款、付款、换汇、融资、对账都在 WorldFirst 里发生，WorldFirst 就从工具变成经营入口。

L3「换汇」是六个阶段中决策密度最高的一环：它的最优解需要同时调用 L1（多少钱进来）、L4（要付多少出去）、L5（能不能留存）、L6（差额能不能借）。这意味着 L3 的好坏不取决于"换汇页本身做得多好"，而取决于"L1–L6 的信息能不能被一次关联"。这是本方案选择切入 L3 的根本原因。

02 / PAIN POINTS

现状痛点

2.1 现状产品流程 vs 商家真实决策心智

这一节是整份方案的起点。对比的是：当前线上 WorldFirst 兑换页给商家的步骤，与商家心里其实需要走完的步骤之间的差距。

现状产品流程

三步走完，无任何决策辅助

  • 进入兑换页
  • 输入金额
  • 点击「确认兑换」
流程耗时：30 秒。但所有判断由商家自己心算完成。
商家真实决策心智

六件事要想清楚，才敢按下确认

  • 这笔钱换完是用于哪一项经营动作
  • 本月还有哪些同币种付款承诺
  • 本月可结汇额度够不够、凭证齐不齐
  • 当前汇率位置是什么背景
  • 路径成本：直接全换 vs 部分保留 vs 分批换
  • 最终决定换多少、留多少
实际耗时：15–30 分钟，且做完不一定对。

2.2 痛点拆解

对应 2.1 中六件事，归纳出五类高频痛点。「AI 杠杆度」这一列衡量"这件事如果由 AI 来关联和解释，能给商家省多少动作"。

痛点商家原话一年潜在成本（参考 ¥3000 万 GMV）AI 杠杆度
不知道这笔钱该不该全换 "全换还是分批？" 2–5 万 ★★★★★
不知道留外币还是换人民币 "下月广告费要不要留美元？" 4–10 万 ★★★★★
撞到结汇额度不足才发现 "为什么换不了？" 时间 + 延误付款 ★★★★

决策后无反馈 "这次换得对吗？" 长期 ★★★

2.3 痛点的本质：信息都在 WorldFirst，但没被关联

2.2 表面上看起来是"商家不会判断"，但拆开来看是另一种结构——决策需要的信息其实都已经在 WorldFirst 后台，只是分散在五个不同的位置：

决策需要的信息当前所在位置
多币种余额首页 · 我的资产
平台回款时间账单管理中心
历史付款记录（含广告 / SaaS / 海外仓）全球付款 · 历史记录
当前与历史汇率首页 · 实时汇率
可结汇额度 + 凭证完整性店铺管理 · 额度管理

核心判断：问题不是"信息不在一处"，而是"信息在一处但需要商家自己开五个 tab、自己心算关联"。AI 的价值不是搬运信息，是关联。

03 / GOALS

需求目标

3.1 北极星指标

MVP 观察目标：有效决策完成率 40%+

有效决策定义：用户在卡片浮现后完成以下任一动作——按建议部分兑换 / 保留外币 / 设置汇率提醒 / 跳转补凭证 / 在确认风险后继续原路径。

这个指标的设计逻辑是避免"卡片采纳率"这种容易诱导强推的单一指标。"在确认风险后继续原路径"也算有效决策——只要用户被帮助到，哪怕选择"我什么都不改"也算成功。

3.2 反向兜底硬约束

北极星不能孤立看，必须配反向指标兜底，否则容易被刷指标。任一项超过阈值，暂停灰度，复核 prompt 与策略规则。

指标阈值含义
错误推荐率< 3%财务师月抽审 100 笔，标注错误推荐
用户取消执行率< 8%执行后 5 分钟内主动撤销的比例
卡片关闭率< 50%卡片浮现后用户直接点 × 关闭的比例（高则代表打扰）

3.3 业务效果指标

北极星衡量"用户被帮助到"，业务指标衡量"WorldFirst 因此获得了什么"。

指标计算口径

二次换汇笔数减少同一商家 30 天内"先 USD→CNH 又 CNH→USD"的笔数变化

04 / SCENARIOS

用户场景与问题拆解

4.1 触发场景与触发条件

本方案只在一个场景下被触发：商家在「资金管理 → 货币兑换」页输入金额后。其他页面 MVP 阶段不触发，避免范围扩散。

条件类型规则
必须满足金额 ≥ 1 万 USD 等值，且距离上次同币种兑换 > 24 小时
不触发小额（< 1 万）、高频用户（24h 内已兑换）、用户已主动关闭过卡片

4.2 四类用途场景

系统根据余额、历史付款记录、收款人类型、本次金额，自动识别四类用途场景。每类场景对应一种 AI 介入方式（详细介入策略见 08 章）。

场景 A · 付供应商

用途明确，不需要 AI 多说话

"我要付国内货款，今天换多少美元？"

识别条件：历史付款偏向 RMB、收款人为国内供应商、金额在历史合理区间。
用户问题：换多少够付？额度撑得住吗？凭证全吗？

场景 B · 保留外币付款

会犯"二次换汇"错误的高频场景

"我有美元，下月还要付广告/SaaS，是不是该全换？"

识别条件：账上有外币余额、近 30 天有同币种付款记录、本次为大额结汇。
用户问题：全换会不会下月再换回来？保留多少够？

场景 C · 暂时不用

没有明确用途，最容易被错误引导

"这笔外币暂时没用处，全换还是留？"

识别条件：用户主动选择"暂时不用"，或历史无明确未来付款承诺。
用户问题：现在汇率是高是低？分批好不好？要不要锁定？

场景 D · 额度不足

合规硬约束被动撞墙

"我想换，但为什么换不了？"

识别条件：本次结汇金额超过本月可用额度，或检测到店铺有未补凭证的订单。
用户问题：到底缺哪些凭证？补完要多久？这笔款延误了怎么办？

4.3 不做什么（MVP 边界）

明确划出 MVP 范围之外的事

  • 不预测汇率走势——属金融预测，监管风险高
  • 不做主动推送（首页推 / push）——Phase 2 再做，先验证被动浮现的卡片有效性
  • 不扩展到付款页——Phase 2 再做，先把兑换页一个场景做透
  • 不做现金流预测、ERP 集成——Phase 3，依赖外部数据源
  • 不做 AI 自动执行——永远不做，资损与监管红线
05 / SOLUTION

方案概述

5.0 关键认知升级：从「汇率驱动」到「用途驱动」

本方案在迭代过程中经历了一次关键认知调整。早期版本的产品定位是"在兑换前帮商家判断该不该换"，本质是"汇率驱动"。后来发现这条路不通——一旦走"判断汇率"的方向，就会同时踩到三个坑：1）金融预测的合规边界；2）与银行 quant 工具的同质化；3）和商家的真实决策路径不一致。

真正的转折点是这个观察：商家换汇从来不是为了交易汇率本身，而是为了后续的某个经营动作。由此把策略输入从"汇率位置"改为"经营用途"——这是 WorldFirst 才能做、且银行 quant 工具做不了的事。

5.1 由此得出的三条策略

策略 01

用途驱动，不是汇率驱动

策略输入的起点是"这笔钱换完做什么"，由系统根据历史付款记录主动推断，用户可一键修改。AI 不预测汇率走势，仅引用当前汇率所处的历史区间作为背景。

策略 02

关联现有数据，不新增数据源

所有策略输入仅来自 WorldFirst 内部高确定性数据：余额、历史付款、可结汇额度、凭证状态、历史汇率。MVP 阶段不依赖任何外部数据（如 ERP、广告后台、税务日历）。

策略 03

三不原则：可审计的边界

不预测汇率、不承诺收益、不自动执行。金额拆分、路径成本、额度校验由可审计的规则引擎完成；LLM 只负责解释和交互。

5.2 产品定位

AI 是辅助决策层，不是汇率预测工具，也不是金融产品销售。所有预测能力被刻意阉割，所有执行权保留给用户。这条主线决定了下文每一个设计点的边界。

06 / WORKFLOW

主链 · 第 1 步 · AI 折叠卡浮现

用户在兑换页输入金额并失焦后，卡片以折叠态浮现：仅显示一行"预计用途 + 推断依据"，不打断兑换主流程。用户可直接关闭。

主链 · 第 2 步 · 展开建议方案

用户点击「查看建议方案」后展开三步建议（兑换金额、保留金额、辅助动作如汇率提醒）。同时显示"当前路径 vs 建议路径"的成本对比，措辞统一为"减少路径成本"，不使用"节省"。

主链 · 第 3 步 · Deep link 执行

每个建议项都对应一个 deep link 按钮，点击后跳转到目标功能页（货币兑换 / 汇率提醒 / 安心兑 / 店铺管理-额度），参数预填，但用户必须二次确认才会真正提交。

主链 · 第 4 步 · 即时路径对比 + 反馈采集

执行后即时显示"按当前费率估算可减少路径成本约 ¥XXX"。基于决策当时的可选路径对比，不依赖未来汇率走势的事后回算。同时采集即时反馈（👍 / 👎 / 关闭原因）作为后续策略优化输入。

支线 · 追问对话（可选）

用户在第 1 步看到推断或第 2 步看到方案后，若对依据或建议有疑问，可点击「追问 AI」打开右侧对话抽屉。AI 已在上下文中获知当前推荐内容，用户可点击预设问题或自由提问；可基于用户新信息（如"广告投放计划取消"）即时调整建议，回到第 2 步重新展示新的方案。

07 / CAPABILITY

产品能力说明

支撑核心产品链路的五层能力。所有数字相关计算由后端规则引擎完成，LLM 只承担文案组装和对话交互。

能力 01 · 触发引擎

判断什么时候浮现卡片

金额阈值校验、用户分群、频次控制、关闭历史记忆。决定不同用户在同一动作下是否看到卡片。

能力 02 · 推荐策略引擎

识别用途、计算金额、生成方案

聚合余额 / 历史付款 / 可结汇额度 / 凭证状态等内部数据，识别四类用途场景，输出"该换多少 / 该留多少 / 是否触达额度风险"。所有数字由规则计算，无 LLM 介入。

能力 03 · LLM 组装层

把规则结果翻译成自然语言

按场景模板填空生成可解释文案；追问对话支持 Function Calling 调内部 API 改建议；System prompt 注入三不原则与措辞红线。

能力 04 · Deep link 执行层

把建议转成可执行动作

每个建议项对应一个 deep link，跳转目标功能页时预填参数；用户二次确认后才真正提交；执行后回写实际操作金额，作为后续优化依据。

能力 05 · 反馈与追踪层

关闭决策闭环

即时反馈（👍 / 👎 / 关闭原因）+ 执行后路径成本展示 + 月度财务师抽审 100 笔。错误推荐率超过 3% 暂停灰度。

08 / KEY DESIGN

关键设计点

D1–D4 按四类用途场景分别设计 AI 介入方式；D5–D6 是贯穿所有场景的纪律。介入的强弱由"用户决策的不确定性"和"潜在风险"决定，不由模型置信度决定。

D1 · 付供应商

静默校验 · 不打断

识别条件：历史付款偏向人民币、收款人为国内供应商、本次金额在历史区间内、额度与凭证均正常。

AI 介入：仅在「确认兑换」按钮上方显示一行"已校验：额度充足、凭证完整"绿色提示。不弹大卡，不打断流程。

为什么不弹卡：用户目标已明确，AI 再问「你要干嘛」是负价值。仅在异常情况（如金额显著超过历史均值、额度即将不足）才升级为弹卡。
D2 · 保留外币付款

主动建议 · 浮现完整方案

识别条件：账上有外币余额、近 30 天有同币种付款记录（广告、SaaS、海外仓）、本次为大额结汇。

AI 介入：浮现完整 AI 卡，推断用途为"保留外币付款"，附依据（"近 30 天 X 笔同币种付款，共 $XX,XXX"），给出"兑换部分 + 保留部分 + 设汇率提醒"三步方案 + 路径成本对比。

为什么主动建议：这是用户最容易"全换了又换回来"产生二次换汇成本的场景。AI 的关联价值在此最大。
D3 · 暂时不用

克制建议 · 不交叉销售

识别条件：用户主动选择"暂时不用"，或历史无明确未来付款承诺。

AI 介入：仅推荐"分批兑换 + 汇率提醒 + 暂不兑换"三个低风险动作。金融产品（万余盈、安心兑）入口仅在用户已有持币 30 天以上 / 已使用过该产品 / 主动选择"暂时不用"等触发条件下才显示。

为什么克制：避免"用户不知道做什么"被动转化为"系统推销金融产品"。匹配条件不满足时不展示金融产品入口。
D4 · 额度不足

异常拦截 · 引导补凭证

识别条件：本次结汇金额超过本月可用额度，或检测到店铺有未补凭证的订单。

AI 介入：浮现警示色卡片，明确告知"本次最多可结汇 X 美元"和"X 笔订单缺凭证"，主按钮直接跳转到「店铺管理-额度管理」预填到那几笔订单。

为什么必弹：境内合作伙伴展业三原则要求人民币结汇必须有真实贸易凭证。这是合规硬约束，让用户撞上去再补，不如提前拦下来。

D5 · 数据边界（贯穿全场景）

MVP 阶段，所有策略输入仅来自 WorldFirst 内部高确定性数据：余额、历史付款、收款人类型、可结汇额度、凭证状态、当前与历史汇率。未来付款需求通过历史模式和已创建付款草稿推断。不接入任何外部数据源（ERP、广告后台、VAT 日历），避免假设过多导致不可控。

D6 · 措辞红线（贯穿全场景）

不能用改成
多赚 ¥X / 节省 ¥X减少路径成本约 ¥X
美元将上涨 / 下跌当前处于近 30 天 XX 区间（参考）
推荐你立即买入你可以考虑设置汇率提醒
本操作可获得收益本操作可降低成本

所有金额数字由后端规则引擎生成，LLM 在 prompt 层面被明确禁止预测、计算、承诺收益。措辞规则在 system prompt 里以红线形式注入。

09 / USER CASE

用户 case · 一次保留外币的决策

CASE WALKTHROUGH 小王的周一上午
小王是一名多平台跨境电商卖家，月 GMV 约 50 万美元。周一上午，他在 WorldFirst 收到一笔 5 万美元的 TikTok Shop 平台回款，原本准备全部兑换成人民币，用于国内备货和日常周转。

当他进入「货币兑换」页并输入 50,000 USD 时，系统识别到：过去 30 天内他有稳定的 USD 广告和 SaaS 付款记录；同时本次的人民币付款需求并不需要全额覆盖。

于是页面在「确认兑换」按钮上方浮现一张「AI 资金建议」卡，建议他只兑换其中一部分，保留部分美元用于后续外币付款，减少不必要的二次换汇路径成本。
STEP 1 浮现 看到推断 卡片显示「预计用途：保留外币付款」「依据：近 30 天 5 笔 USD 付款，共 $32,100」
STEP 2 展开 看到方案 三步建议——兑换 20,000 USD、保留 30,000 USD、设置 USD/CNH 汇率提醒；并展示当前路径 vs 建议路径成本对比
STEP 3 追问 问清楚 小王不确定为什么是 30,000，点击「追问 AI」，系统答："这是按你过去 30 天 USD 付款总额推算的下月需求估算"
STEP 4 执行 一键应用 点击「一键应用建议」，金额自动改为 20,000 USD，并预填了汇率提醒。小王二次确认后提交
STEP 5 反馈 看到结果 底部出现绿色 banner："已应用建议方案 · 减少一次未来二次换汇链路，按当前费率估算可减少路径成本约 ¥XXX"

这个 case 验证了三件事

  • 策略 01 兑现：AI 主动识别"保留外币付款"用途，从历史付款记录中关联出依据，而不是问"你要换汇做什么"
  • 策略 02 兑现：所有数字（5 笔付款、$32,100、30,000）全部来自 WorldFirst 内部数据，没有任何预测
  • 策略 03 兑现：AI 给出建议，但金额拆分由规则引擎完成；执行权由用户二次确认；措辞使用"减少路径成本"，不说节省
10 / RISK

风险评估

以下是已识别的 5 类风险与对应缓解措施。合规与数据是最高优先级风险，模型与工程次之，冷启动最低。

高 · 合规
措辞误踩监管红线
LLM 偶发输出"将上涨""推荐买入"等违禁措辞
缓解：System prompt 注入红线 + 输出后正则扫描 + 财务师月抽审 100 笔
高 · 数据
商家历史数据不完整
新商家或低频用户无足够付款记录可推断
缓解：数据置信度低于阈值时不浮现卡片，仅做静默校验
中 · 模型
LLM 偶发跑偏或过度迎合
追问对话中可能虚构数据或被用户引导突破红线
缓解：金额、汇率全部由规则引擎注入，LLM 不参与计算；红线提示重复出现在 system prompt
中 · 工程
第三方 LLM API 网络波动
面试、演示、生产环境均可能出现 API 不可用
缓解：对预设高频问题准备本地 fallback 回答；触发 fallback 时在 UI 标注
低 · 冷启
新用户无任何历史信息
无法识别用途、无法做关联
缓解：冷启商家仅触发静默校验和额度拦截，不做主动建议
